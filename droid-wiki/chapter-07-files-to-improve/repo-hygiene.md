# F — Repo Hygiene

> Three issues at the repository level. None are huge LoC numbers but each shapes day-to-day clone/edit/CI experience.

---

## Binary in repo

**Path:** `cli/vllm-studio`
**Size:** **60,308,320 bytes (~60 MB)** — verified `ls -la cli/vllm-studio`.

### Symptoms

- A 60 MB compiled CLI binary is checked into git.
- Every clone pays 60 MB of unnecessary bandwidth.
- Every CI cache pays 60 MB.
- The binary is regenerated by build, so the committed copy is **always wrong** the moment anyone changes `cli/src/`.
- It also bloats `git log` — `git pack` becomes painful at scale.

### Proposed action

Three changes, in order:

1. **Add to `.gitignore`** at the repo root:
   ```
   /cli/vllm-studio
   ```
2. **`git rm --cached cli/vllm-studio`** so it is removed from tracking but kept on disk.
3. **CI release:** wire the binary build into the release workflow. The release config already exists (`release.config.cjs`). Add a job that:
   - builds the binary (`cd cli && npm run build:bin` or whatever the existing recipe is — verify by reading `cli/package.json`),
   - uploads the artifact to the GitHub release,
   - never commits it back.

Optional next step (separate PR): use `git filter-repo` to expunge the binary from history. This is **strongly recommended** if the team is comfortable with a force-push, because the 60 MB blob otherwise lives forever in `.git`. Coordinate with everyone with active branches first.

### Estimated impact

- Disk delta on a fresh clone: **–60 MB**.
- LoC delta: 0.
- Risk: **low** to ignore + cache-rm; **medium** if history is rewritten (force-push consequences).

### Dependencies

- None for steps 1–2. Step 3 depends on the existing CI workflow; if no release CI exists, step 3 is a separate workpack.

## Root design-doc overlap

**Files at repo root:**

- `MIGRATION.md` — the migration history (Phases 1–5 of controller restructuring).
- `scope.md` — the Pi-agent integration scope (frontend-heavy, full feature plan).
- `plan.md` — referenced in the task brief; verify presence with `ls plan.md`.

### Symptoms

- Three living design documents at the repo root with overlapping subject matter.
- `MIGRATION.md` is fine where it is — it's a status log.
- `scope.md` is forward-looking — it should live with the rest of the docs.
- `plan.md` (if present) is harder to place without inspection — it may be obsolete.
- See [Chapter 4 — root docs and plans](../chapter-04-anything-else/root-docs-and-plans.md) for the full inventory.

### Proposed action

| Doc | Action |
|-----|--------|
| `MIGRATION.md` | **Keep at root.** This is the canonical history. |
| `scope.md` | Move to `docs/scope/pi-agent-integration.md`. Add a status banner at the top: `> Status: planning. Source-of-truth for Pi-agent integration. Last updated: <date>.` |
| `plan.md` | Read it. If it's superseded by `scope.md` or `CONTROLLER_SCOPE.md`, **delete it**. Otherwise move to `docs/plan/<short-title>.md` with a status banner. |
| `CONTROLLER_SCOPE.md` | **Keep at root.** It's the controller's design contract. (Not flagged for change here.) |

After this, the repo-root design docs are exactly two: `MIGRATION.md` (history) and `CONTROLLER_SCOPE.md` (forward-looking spec for the controller). Everything else moves into `docs/`.

### Estimated impact

- LoC delta: small (mostly file moves).
- Risk: **none** — docs only.
- Discoverability: significantly improved — `docs/` becomes the place to look.

### Dependencies

- Update any `README.md` cross-references after the move.

## Duplicated state-machine

**Files:**

- `frontend/src/lib/state-machine.ts` — **45 LoC** (verified).
- `controller/src/modules/shared/state-machine.ts` — **45 LoC** (verified).

### Symptoms

- Identical (or near-identical) generic state-machine helper duplicated across two packages after the `shared/` package was dissolved (see [Chapter 4 — shared package dissolution](../chapter-04-anything-else/shared-package-dissolution.md)).
- Verified by reading both: same `StateMachineTransitionResult`, `StateMachineTransition`, `StateMachineContainer` shapes.

### Proposed action

Pick one, in preference order:

1. **Accept duplication as deliberate isolation.** Two ≈45-line files is fine if package isolation is the goal — the cost of a tiny shared package is higher than the cost of two copies. **Document this decision** with a comment in *both* files:
   ```ts
   // Note: this file is intentionally duplicated in
   // <other path>. Keep them in sync by hand.
   ```
   This is the recommended option for a 45-LoC helper.

2. **Republish via package.** Create `packages/state-machine/` (or `tools/state-machine/`) — workspace package, single tsconfig, single export. Update both consumers.
   - Pro: single source of truth.
   - Con: workspace plumbing for ~45 LoC.
   - Only worth it if other small helpers want to follow.

3. **Inline into each consumer.** Delete the helper files; copy the functions into whichever 2–3 call sites in each package use them. Eliminates the abstraction. Only viable if each consumer touches a small number of files.

Recommended: **option 1** unless multiple other helpers want to follow into a shared package.

### Estimated impact

- LoC delta: 0 to ~–45 depending on option chosen.
- Risk: **low** in all cases.

### Dependencies

- This is a **decision task**, not a code task. Pick one, document it, move on.

## Summary

| Item | Action | Disk / LoC delta | Risk |
|------|--------|------------------|------|
| `cli/vllm-studio` binary | Gitignore + release-via-CI | **–60 MB** | Low |
| `MIGRATION.md` / `scope.md` / `plan.md` overlap | Keep `MIGRATION.md` at root; move/delete others | small | None |
| `state-machine.ts` duplication | Document the duplication (option 1 recommended) | 0 | Low |

The binary removal alone is the biggest single win in this whole chapter for **clone latency** and CI cache size.

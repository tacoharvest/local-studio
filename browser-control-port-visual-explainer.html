Browser control port plan

Parchi-style browser reliability without a Parchi-sized rewrite

This plan keeps the current HTTP → SSE → Electron <webview> bridge and adds a thin, testable command layer. The goal is better browser control with less inline JavaScript in agent-workspace.tsx, not a second automation architecture.

Prompting Change

Minimal version: prepend a browser-control instruction block to the user message only when browser tools are enabled. This avoids changing pi internals while giving the model Parchi-like discipline. 

<browser_control>
When using browser tools:
1. Navigate or read the current URL before interacting.
2. Prefer browser_wait_for over guessing page readiness.
3. After click/fill/press/scroll, verify with browser_get_content.
4. Use browser_evaluate for batch DOM extraction or repeated actions.
5. If a selector fails, try text=..., wait_for, then evaluate.
</browser_control>

Expected LOC if implemented in chat-pane.tsx request assembly: +22. I would keep this optional if we want zero changes to prompt semantics in the first pass.
